一、项目总体定位

项目定位为教学综合信息服务平台，复刻学校教务系统中的核心场景。它不是单纯展示页面，而是同时包含学生、教师、管理员三类角色，以及登录认证、权限菜单、课程选课、成绩考试、通知评价、文件材料和审计日志等业务闭环。

学生角色

课程查询、自主选课、抢课、退课、个人课表、空闲教室、成绩查询、考试查询、通知查看、教学评价。

教师角色

任课课程、成绩查看、考试安排查看、教学评价结果查看，形成教师端教学工作台。

用户、角色、权限、菜单、成绩、考试、课程、通知、评价、审计日志、文件导入导出和申请材料管理。

整体技术路线

后端

Spring Boot、Spring MVC、Spring Security、MyBatis、JPA、Bean Validation、Flyway、Redis、H2/MySQL。

前端

Vue 3、TypeScript、Vite、Vue Router、Pinia、Axios、Element Plus、ECharts、Lucide 图标。

数据

Flyway 建表，DataInitializer 初始化学院、专业、班级、学生、教师、课程、成绩、考试、菜单和权限等演示数据。

接口

统一使用 ApiResponse<T>；分页数据使用 PageResponse<T>。

一句话总结：项目用 Spring Boot + Vue 3 + Redis 完成一个接近真实教务系统的综合业务复刻，核心价值在于角色清晰、流程完整、权限可控、Redis 有真实使用场景。

二、魏语石：系统骨架、认证权限与后台基础

魏语石负责项目总体架构、登录认证、权限体系、后台管理、公共响应结构、异常处理、审计日志、菜单体系和前端整体布局。他的模块让整个系统“能登录、能鉴权、能按角色展示菜单、能管理后台”。

登录认证

实现账号密码登录、BCrypt 密码校验、登录失败次数限制、访问令牌、刷新令牌和退出登录。

登录页提交前端调用 loginApi。
服务端校验AuthService 查询用户并使用 BCrypt 校验密码。
生成双 token生成 access token 与 refresh token。
Redis 存储AuthTokenStore 写入 Redis，并保留本地内存 fallback。
前端入站Pinia 保存 token，路由进入主界面。

权限与安全控制

通过 BearerTokenAuthenticationFilter 解析 token，查询用户角色与权限，结合 @PreAuthorize 保护关键接口。

  • 管理员接口必须具备管理员角色。
  • 成绩、课程、状态审核等接口通过权限码控制。
  • 前端根据后端菜单生成可见入口。
  • Vue Router 负责未登录拦截。

后台用户管理

管理员可以查看用户、创建测试用户、重置密码、分配角色。后端用 Spring MVC + MyBatis，前端用 Element Plus 表格、表单和弹窗。

操作日志与审计

记录教学班修改、申请审核、公告发布、角色权限调整等关键操作，便于追踪谁在什么时候修改了什么。

三、郭凤圣：学生选课、抢课、课表与基础数据

郭凤圣负责学生端课程业务，包括课程查询、自主选课、抢课、退课、个人课表、空闲教室和课程基础数据展示。这部分是项目中最能体现业务闭环和 Redis 高并发价值的模块。

自主选课与退课

实现可选课程列表、已选课程列表、选课、抢课入口、退课和选课时间窗口判断。

  • 学生进入自主选课页面。
  • 前端调用课程列表接口。
  • 后端按学生账号和学期查询可选课程。
  • 选课时校验身份、时间窗口、重复选课和容量。
  • 成功后写入选课记录，并刷新余量。

抢课模块与可替换边界

抢课模块抽象为 CourseGrabPort，当前真实实现是 LocalCourseGrabService。Controller 依赖接口而不是具体实现，后续如需拆分服务可以替换实现。

  • 课程列表和已选列表已加入分页。
  • 前端抢课时生成 requestId。
  • Redis 负责库存、短锁、幂等。
  • @Transactional 保证最终落库一致性。

个人课表与空闲教室

通过 MyBatis 按学生、班级、学期、周次查询课程数据，前端把课程渲染成周一到周五的课表格子，并支持空闲教室查询。

数据初始化

补充学院、专业、班级、学生、课程、课表、成绩和考试安排等演示数据，覆盖信息科学与工程学院、医学院、经济管理学院、艺术学院。

四、敖东磊：教师端、成绩考试、通知评价与文件

敖东磊负责教师端、成绩管理后台、考试管理后台、通知公告、教学评价、文件导入导出和部分学生申请审核页面。他的模块让教师和管理员能够维护教学过程中的结果数据。

教师端

教师登录后可查看任课课程、成绩、考试安排和教学评价结果。后端从认证上下文获取用户名，再按教师姓名查询业务数据。

成绩管理后台

支持成绩查询、新增、修改、锁定、补考/重修/缓考状态、CSV 导入导出。查看由 GRADE_READ 控制，录入修改由 GRADE_WRITE 控制。

考试管理后台

维护考试安排、考场、座位和监考信息。新增或修改考试后，系统查询选中该课程的学生并生成通知。

通知、评价与文件

实现首页公告、分类通知、角色定向通知、已读未读状态、教学评价统计、课程/成绩 CSV 导入导出和学籍申请材料上传。

Redis 使用位置与技术意义

Redis 在项目中分成三类：认证状态、抢课保护、后台维护后的库存缓存一致性。这里特别区分“已实现落点”和“后续可扩展点”。

1

auth:access:{token}
保存 access token 对应的用户名，适合短期访问令牌校验。

2

auth:refresh:{token}
保存 refresh token 对应的用户名，用于刷新访问令牌。

3

auth:failures:{username}
保存登录失败次数，超过阈值后临时锁定账号。

对应代码：AuthService、AuthTokenStore。技术意义是 token 和失败次数都有 TTL，Redis 能自动过期，避免频繁写数据库。
1

selection:offering:{offeringId}:remaining
课程剩余容量，用于抢课时快速扣减库存。

2

selection:grab:lock:{offeringId}:{username}
同一学生同一课程的短期处理锁。

3

selection:request:{requestId}
抢课请求幂等 key，防止重复点击造成重复提交。

对应代码：LocalCourseGrabService#reserveStock、#tryLock、#markRequestProcessing、#cacheRequestResult。数据库仍是最终数据源，Redis 是高并发保护层。
1

selection:offering:{offeringId}:remaining
管理员修改教学班容量、教室、时间等信息后，需要清理抢课库存缓存，下一次抢课重新根据数据库计算余量。

对应代码：AdminCourseController#evictOfferingStock、LocalCourseGrabService#ensureStockKey、#stockKey。

魏语石：Redis 库存扣减检测

负责课程剩余名额缓存、并发扣减、防超卖，以及数据库写入失败后的 Redis 库存回滚。

key

selection:offering:{offeringId}:remaining
每个教学班一个库存 key，值是当前剩余名额。

  • 抢课前按数据库容量和已选人数预热库存。
  • 高并发抢课时先扣 Redis 库存。
  • 库存不足时快速返回 FULL。
  • 数据库写入失败时补回 Redis 库存。
  • Redis 监控页展示数据库容量、已选人数和是否超卖。

郭凤圣：Redis 请求幂等检测

负责防止同一次抢课请求重复提交，处理前端重复点击、网络重试和同一个 requestId 再次提交。

key

selection:request:{requestId}
一个抢课请求对应一个幂等 key，用来记录处理中或已完成结果。

  • 前端每次抢课生成 requestId。
  • 后端先检查 requestId 是否已经处理。
  • 重复请求直接返回缓存结果，不重复写数据库。
  • 选课页面已加入分页，方便大数据量展示。
  • Redis 监控页可查看幂等 key 是否存在和 TTL。

前端入口层
Vue 3
浏览器访问学生、教师、管理员进入不同业务页面。
Vue Router登录守卫判断 token 和动态菜单。
Pinia 状态保存用户、角色、权限、菜单和登录态。
Axios 请求统一携带 Bearer Token 调用后端接口。
HTTP / JSON / Bearer Token
后端业务层
Spring Boot
Security Filter解析 token，构造认证上下文。
Controller接收登录、选课、成绩、考试、通知等请求。
Service完成业务校验、事务控制、日志和通知生成。
Mapper / Repository通过 MyBatis、JPA 访问业务数据。
事务数据 / 缓存状态 / 文件资源
数据资源层
DB + Redis
MySQL / H2保存用户、课程、成绩、考试、通知等最终数据。
Redis保存 token、登录失败次数、抢课库存、短锁和幂等结果。
Flyway版本化维护数据库结构和演示数据基础。
uploads保存学籍申请、材料上传等文件资源。
答辩重点：前端负责交互，后端负责权限和业务，数据库是最终数据源。
Redis 不是装饰：它承担登录态、失败次数、抢课库存和幂等保护。
所有模块统一返回 ApiResponse / PageResponse，便于前后端协作。
