# v1.4 Additional Software Test Findings

Updated: 2026-06-27

This document records additional QA findings after reviewing the latest Academic-Nexus repository state. The goal is not to add brand-new modules, but to identify broken role/data/status/audit closures in existing features.

## 1. AI model selection does not truly propagate to ai-service

Finding:

- `AiChatView` sends `modelId` through `sendAiChatMessageApi(sessionId, { message, modelId })`.
- `AiChatSessionService.send()` resolves the selected model and calls `chatService.chat(userText, principal, model.id(), sessionId)`.
- `AiChatService.chat()` resolves `selectedModel`, but then calls `remoteClient.chat(message)`.
- `AiRemoteClient.chat(String message)` sends only `{ message }` to ai-service.
- `AiServiceFeignClient.ChatPayload` only contains `message`.
- `ai-service` `AiDtos.ChatRequest` only contains `message`.
- `AiInternalController.chat()` always calls `ollamaClient.generate(chatModel, prompt)`, where `chatModel` is the configured default.

Risk:

User selects model A, but ai-service still calls default qwen3:8b. The UI then shows a misleading result where the selected model and actual model diverge.

Required fix:

- Extend main-system `AiRemoteClient.chat` to accept target modelName/modelId.
- Extend Feign `ChatPayload` to include `modelName` or `targetModel`.
- Extend ai-service `AiDtos.ChatRequest` to include optional `modelName`.
- In `AiInternalController.chat`, use requested model if present and allowed, else fallback to configured chatModel with explicit fallback reason.
- Return both expectedModel and actualModel if fallback occurs.
- AI call log must record selected model, actual model, and fallback reason.

Suggested regression test:

- Create two enabled CHAT models.
- Send chat with non-default modelId.
- Assert response.modelName equals selected model when ai-service can use it.
- Assert fallback is explicit if ai-service cannot use it.
- Assert ai_call_log contains selected modelId and actual model.

## 2. Sensitive words / content moderation page 500 and uncaught mounted hook

Finding:

- `/admin/sensitive-words` page loads `sensitiveWordsApi` and `moderationLogsApi` in a single `Promise.all`.
- If either endpoint returns 500, `loadData()` has no catch block, causing uncaught Promise in mounted hook.
- Backend `GovernanceController` directly queries `sensitive_word` and `content_moderation_log`; unlike `ContentModerationService`, it does not catch `DataAccessException` or fallback to empty pages.

Required fix:

- Backend must return empty paginated results or readable 4xx/5xx with message; not raw 500 for missing/drifted governance tables.
- Frontend `SensitiveWordsView.loadData()` must catch errors and load the two sections independently, so word list failure does not blank log list and vice versa.
- Add visible error alert for each failed section.

Suggested regression test:

- ADMIN GET `/api/admin/sensitive-words?page=1&size=50` returns 200.
- ADMIN GET `/api/admin/content-moderation/logs?page=1&size=20` returns 200.
- STUDENT/TEACHER get 403.
- Empty table returns empty page.
- Frontend page handles one API failure without uncaught mounted hook.

## 3. Load test panel reads zero offerings through backend API

Finding:

- `course_grab_panel.py` defaults to backend API mode.
- It calls `/api/admin/course-offerings?page=1&size=200`.
- The front-end admin course-offering API uses `/admin/course-offerings` with optional `term`, returning an array, not necessarily a paginated `{records}` shape.
- The script parses `payload.data.records`, so if the backend returns an array, it reads zero rows.
- The parser also uses `item.get("id")`, while frontend DTO uses `offeringId`.

Required fix:

- Make the Python parser support both response shapes:
  - `data.records[]`
  - `data[]`
- Accept both `id` and `offeringId`.
- Include `term` option in the panel, default `2025-2026-2`.
- Print clear diagnostic: HTTP status, data shape, record count, first record keys.
- If backend returns empty list, suggest checking term, demo data, auth token, and selection window.

Suggested regression:

- Add a small unit test for `course_grab_panel.py` response parsing with both API shapes.
- Add a backend smoke test for `/api/admin/course-offerings?term=2025-2026-2` returning non-empty demo data under demo profile.

## 4. Batch operations are documented but not fully closed in UI/API

Finding:

- `BatchTasksView` is currently a list page with taskType/status filters and report path text.
- `batchOps.ts` exposes only list API for batch tasks, and data-archive APIs.
- `UserAdminView` still only supports single-user CRUD, role assignment, password reset, delete.
- `adminUser.ts` has no batch import/export/precheck APIs.
- `adminCourse.ts` has no batch course/offering import/precheck APIs.
- Registration and status-change admin pages still use single-row approval/rejection.

Required closures:

- Batch user import/export with precheck and report.
- Batch course/offering import with precheck and Redis stock consistency handling.
- Batch review for status changes and registration applications.
- Batch task detail/report download instead of reportPath as text only.

## 5. Teacher-side awareness gaps

Finding:

- Status-change flow is student + admin. Homeroom teacher cannot see status-change state for their own class.
- Registration application flow is student + admin. Course-related applications do not appear in teacher side.

Suggested closure without creating new large modules:

- In teacher homeroom class page, add readonly status-change/application summary badges for students in their class.
- In teacher course page or a small readonly tab, show course-related registration applications for teacher-owned offerings.
- Teachers must not approve unless explicitly required; readonly awareness is enough.

## 6. Grade audit lacks explicit old/new value visibility

Finding:

- Teacher grade page has a reason field for existing grade updates.
- Admin grade page supports edit/import JSON.
- The QA report claims `GRADE_UPDATE`, but the UI does not expose oldScore/newScore change history.

Required closure:

- Grade update audit should record old score, new score, old status, new status, reason, operator, traceId.
- Admin grade detail should show recent grade change history or link to filtered audit.
- Student grade page should refresh after grade updates and only show released/published grades according to policy.

## 7. Notification closure is still mostly manual announcements

Finding:

- Admin notice page supports role-based publishing and stats.
- It does not currently show targeting by grade/major/class/offering.
- Business events such as application review, grade publish, exam change, and batch import are not clearly linked to generated notifications.

Required closure:

- Support target range checks or explicitly document current role-only limitation.
- Application review should notify student.
- Grade publish/update should notify student.
- Exam create/update should notify affected students.
- Notification stats should count target/read/unread correctly.

## 8. Frontend error-state coverage remains inconsistent

Finding:

- `SensitiveWordsView` has an uncaught mounted hook when `Promise.all` fails.
- Some student pages like grade/exam/schedule set loading but do not catch and display errors.
- Global axios handles 401/403 redirect, but page-level 500/network failures still need visible error states.

Required closure:

- All mounted data-loading pages should catch and display readable errors.
- Empty state and error state must be distinguishable.
- File download/API failure must not silently fail.
