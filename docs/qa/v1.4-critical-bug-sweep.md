# v1.4 Critical Bug Sweep

Updated: 2026-06-27

This note records a narrower pass focused on the most severe bugs: authentication/session validity, hard-coded term behavior, post-review evidence mutation, course schedule parsing, and invalid request handling.

## Critical 1: Auth refresh/logout/token invalidation inconsistency

Evidence:

- `AuthService.login()` returns both `accessToken` and `refreshToken`.
- Frontend `auth.ts` and `auth store` persist `refreshToken`.
- `SecurityConfig` permits `POST /api/auth/refresh`.
- `AuthController` does not implement refresh.
- `AuthController.logout()` only returns success and does not revoke current access token.
- `BearerTokenAuthenticationFilter` resolves a token owner and authenticates the user, but does not check whether the account status is still ACTIVE.

Risk:

- Expired access token cannot refresh despite a refresh token being returned.
- Logout may not invalidate old access token.
- Disabled users may continue to use an already issued token until expiry.

## Critical 2: Student status-change attachments can be modified after review

Evidence:

- `StatusChangeAttachmentController.upload()` only calls `ensureOwned(applicationId, principal)`.
- It does not check whether the application status is still SUBMITTED or UNDER_REVIEW.
- Therefore a student may attach additional evidence after the admin has approved or rejected the application.

Risk:

- Audit/evidence chain is broken.
- Admin review result no longer corresponds to the exact materials available at review time.

## Critical 3: Course selection current term is hard-coded

Evidence:

- `CourseSelectionController` uses `private static final String CURRENT_TERM = "2025-2026-2"`.
- Frontend `SelectionView` also displays a hard-coded `2025-2026-2` tag.
- API `courseOfferingsApi` does not accept `term`.

Risk:

- Admin can create/import future terms, but student selection will continue to show only hard-coded term data.
- Load test scripts and demo data may use a different term and appear empty.

## Critical 4: Schedule parsing is fragile

Evidence:

- `PersonalScheduleController.parseSchedule()` parses only strings containing `周二/周三/...` and slot fragments `3-4/5-6/...`.
- Unknown formats silently default to Monday `1-2`.

Risk:

- Imported or batch-created teaching classes with formats such as `星期一`, `Mon`, `第1-2节`, or multiple periods may display on the wrong day/time without error.

## Critical 5: Course grab request lacks required validation

Evidence:

- `CourseSelectionController.CourseGrabRequest` has `Long offeringId` without `@NotNull`.
- If client sends malformed JSON or missing offeringId, backend may pass null into the grab service and produce unclear errors.

Risk:

- Bad requests may turn into internal errors instead of 400.
- Load test tool or front-end request mismatch can become hard to diagnose.
