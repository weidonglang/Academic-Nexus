# 教务管理系统重构开发计划书

## 1. 项目目标

本项目目标是基于 Spring Boot、Vue 3 和 Redis 重构一套现代化教务管理系统。重构重点是复刻教务业务能力、交互流程和数据组织方式，而不是复制原系统代码、静态资源或私有实现。

目标系统面向学生、教师、教务管理员和系统管理员，提供统一身份认证、功能菜单、学籍、选课、课表、成绩、考试、教学评价、毕业设计、通知与系统配置等能力。

## 2. 重构原则

- 功能等价优先：先覆盖学生端核心流程，再扩展教师端和管理员端。
- 架构现代化：前端使用 Vue SPA，后端提供 REST API，避免原系统 iframe/tab 的老式页面结构。
- 数据可追溯：重要申请、审核、成绩、选课、评价等业务必须保留操作日志。
- 权限前置：菜单、接口、数据范围都必须由角色和权限控制。
- 隐私保护：学生身份证号、手机号、成绩等敏感数据按最小权限暴露。
- 可替换集成：预留 CAS/OAuth2、短信、统一支付、教务数据导入等扩展点。

## 3. 原系统抽象分析

原系统是正方教务 V9 风格，主要特征如下：

- 前端技术：jQuery、Bootstrap、jqGrid、iframe tab、服务端渲染 HTML。
- 登录流程：用户名密码登录，密码前端 RSA 加密，后端会话保持，部分场景支持验证码或短信验证。
- 菜单组织：按角色加载功能菜单，学生端菜单包含报名申请、信息维护、选课、信息查询、教学评价、毕业设计等。
- 页面模式：多数功能为查询条件表单 + jqGrid 表格 + 详情/申请/审核弹窗。
- 数据交互：接口多为 `.html` 结尾，实际返回 HTML、JSON 或表格数据。
- 权限模式：通过角色代码、功能模块代码、菜单地址控制访问。

重构时不保留这些技术形态，只保留业务能力和信息架构。

- `evaluation_submission`。

## 8. API 设计规范

统一使用 REST 风格，返回结构：

```json
{
  "code": "0",
  "message": "success",
  "data": {},
  "traceId": "..."
}
```

```json
{
  "records": [],
  "page": 1,
  "size": 20,
  "total": 0
```

- `/api/auth/*`
- `/api/me`

- `/api/schedules/*`

## 10. 安全设计

- 密码使用 BCrypt 存储。
- 登录接口限流。
- 登录失败次数写入 Redis。
- JWT 或 Session 二选一，建议前后端分离使用 access token + refresh token。
- refresh token 存 Redis，支持踢下线。
- 接口使用方法级权限校验。
- 成绩、个人信息等接口强制校验数据归属。
- 审计日志记录登录、修改密码、选课、退课、申请、审核、成绩修改等动作。

## 11. 选课并发设计

选课是系统中最需要 Redis 的场景。

设计方案：

- Redis 保存课程余量 key：`selection:offering:{id}:remaining`。
- 学生提交时先校验登录、批次、规则、冲突、重复选课。
- Redis 原子扣减余量。
- 写入数据库选课记录。
- 数据库写入失败时回补 Redis。
- 定时任务或管理接口对 Redis 和数据库容量做校准。

数据库层仍需唯一约束：

- `student_id + offering_id` 防重复。
- 根据业务需要限制同课程组互斥。

## 15. 开发顺序

当前仓库是 Spring Boot 空骨架，因此先从基础设施开始：

1. 调整后端依赖和配置。
2. 建立统一响应、异常、认证骨架。
3. 建立数据库迁移脚本。
4. 创建 Vue 前端工程。
5. 实现登录和动态菜单。
6. 实现学生首页和个人信息。
7. 再进入选课、课表、成绩等核心业务。

## 16. 风险和边界

- 原教务系统属于商业软件，不能复制其前端资源、代码和私有接口实现。
- 本项目应做业务复刻和体验重建，而不是镜像克隆。
- 如果要接入真实学校数据，需要确认数据授权、隐私合规和网络访问权限。
- 选课和成绩属于高风险业务，必须在测试数据下完成充分验证后再接真实数据。
- Redis 不能作为最终事实来源，数据库仍是权威数据源。
